01 🗄️

What is SQL?

SQL (Structured Query Language) is the standard language for communicating with relational databases. It allows you to create, read, update, and delete data stored in tables — the classic CRUD operations.

💡
SQL is declarative — you tell the database what you want, not how to get it. The database engine figures out the execution plan.

SQL Sub-Languages

CategoryFull NameCommandsPurpose
DDLData Definition LanguageCREATE, ALTER, DROP, TRUNCATEDefine schema structure
DMLData Manipulation LanguageINSERT, UPDATE, DELETE, MERGEModify table data
DQLData Query LanguageSELECTRetrieve data
DCLData Control LanguageGRANT, REVOKEManage permissions
TCLTransaction Control LanguageCOMMIT, ROLLBACK, SAVEPOINTManage transactions

Popular RDBMS Flavors

MySQL PostgreSQL SQL Server (T-SQL) Oracle (PL/SQL) SQLite MariaDB

Core SQL syntax is the same across all, with minor dialect differences in functions and advanced features.

03 🏗️

DDL — Schema Definition

DDL commands define and modify the structure of your database — tables, columns, indexes, constraints.

CREATE TABLE

SQL
CREATE TABLE employees (
    id         INT          NOT NULL AUTO_INCREMENT,
    first_name VARCHAR(50)  NOT NULL,
    last_name  VARCHAR(50)  NOT NULL,
    email      VARCHAR(100) UNIQUE NOT NULL,
    salary     DECIMAL(10,2) DEFAULT 0.00,
    dept_id    INT,
    hire_date  DATE         DEFAULT (CURRENT_DATE),
    is_active  BOOLEAN      DEFAULT TRUE,
    PRIMARY KEY (id),
    FOREIGN KEY (dept_id) REFERENCES departments(id)
        ON DELETE SET NULL
        ON UPDATE CASCADE
);

Common Constraints

PRIMARY KEYUniquely identifies each row. Cannot be NULL. One per table.
FOREIGN KEYLinks to a primary key in another table, enforcing referential integrity.
UNIQUEAll values in column must be distinct (NULLs may be allowed).
NOT NULLColumn must always have a value — cannot be left empty.
DEFAULTProvides a fallback value when no value is specified on insert.
CHECKValidates data against a condition, e.g. CHECK(salary > 0).

ALTER TABLE

SQL
-- Add a new column
ALTER TABLE employees ADD COLUMN phone VARCHAR(20);

-- Modify an existing column's type
ALTER TABLE employees MODIFY COLUMN salary DECIMAL(15,2);

-- Rename a column (MySQL 8+)
ALTER TABLE employees RENAME COLUMN phone TO mobile;

-- Drop a column
ALTER TABLE employees DROP COLUMN mobile;

-- Add a constraint after creation
ALTER TABLE employees
  ADD CONSTRAINT chk_salary CHECK (salary >= 0);

DROP & TRUNCATE

⚡ DROP TABLE

Permanently removes the entire table — structure AND data. Cannot be rolled back (in most DBs).

DROP TABLE employees;
DROP TABLE IF EXISTS employees;

🧹 TRUNCATE

Removes all rows but keeps the table structure. Faster than DELETE. Resets auto-increment.

TRUNCATE TABLE employees;
🚨
Always double-check before running DROP or TRUNCATE in production. These operations are destructive and often irreversible.
04 ✏️

DML — Manipulating Data

INSERT

SQL
-- Single row insert (always name columns explicitly)
INSERT INTO employees (first_name, last_name, email, salary, dept_id)
VALUES ('Priya', 'Sharma', 'priya@company.com', 75000, 2);

-- Multiple rows in one statement (efficient!)
INSERT INTO employees (first_name, last_name, email, salary, dept_id)
VALUES
    ('Arjun', 'Mehta',  'arjun@company.com',  82000, 1),
    ('Sneha', 'Patel',  'sneha@company.com',  68000, 3),
    ('Rohan', 'Gupta',  'rohan@company.com',  91000, 1);

-- Insert from a SELECT (copy data between tables)
INSERT INTO archive_employees
SELECT * FROM employees WHERE is_active = FALSE;

UPDATE

SQL
-- Always use WHERE — otherwise you update EVERY row!
UPDATE employees
SET    salary   = salary * 1.10,
       is_active = TRUE
WHERE  dept_id = 2
  AND  hire_date < '2023-01-01';

-- Update using a subquery
UPDATE employees
SET    dept_id = (
    SELECT id FROM departments WHERE name = 'Engineering'
)
WHERE  email = 'rohan@company.com';
⚠️
Run a SELECT with your WHERE clause before running UPDATE to verify which rows will be affected.

DELETE

SQL
-- Delete specific rows
DELETE FROM employees WHERE is_active = FALSE;

-- Delete with JOIN (MySQL syntax)
DELETE e
FROM   employees e
JOIN   departments d ON e.dept_id = d.id
WHERE  d.name = 'Temp';
05 🔍

SELECT — Querying Data

The SELECT statement is the heart of SQL. It retrieves data from one or more tables.

SELECT Anatomy

SQL
-- Full clause order (you don't need all of them)
SELECT   [DISTINCT] column1, column2, expression
FROM     table_name      -- ① source table(s)
[JOIN    other_table ...]-- ② combine tables
[WHERE   condition]      -- ③ filter rows BEFORE grouping
[GROUP BY column]        -- ④ aggregate rows
[HAVING  condition]      -- ⑤ filter AFTER grouping
[ORDER BY column [DESC]] -- ⑥ sort results
[LIMIT n OFFSET m];      -- ⑦ paginate
💡
SQL processes clauses in this order: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. This affects what you can reference where.

Column Aliases & Expressions

SQL
SELECT
    first_name,
    last_name,
    first_name || ' ' || last_name       AS full_name,
    salary,
    salary * 12                         AS annual_salary,
    salary * 0.10                       AS bonus,
    UPPER(email)                        AS email_upper
FROM employees;

DISTINCT — Remove Duplicates

SQL
-- Get unique department IDs from employees
SELECT DISTINCT dept_id FROM employees;

-- DISTINCT across multiple columns = unique combinations
SELECT DISTINCT dept_id, is_active FROM employees;

CASE — Conditional Logic

SQL
SELECT
    first_name,
    salary,
    CASE
        WHEN salary < 50000  THEN 'Junior'
        WHEN salary < 100000 THEN 'Mid-Level'
        WHEN salary < 150000 THEN 'Senior'
        ELSE                       'Executive'
    END AS salary_band
FROM employees;
06 🎯

Filtering with WHERE

Comparison Operators

SQL
SELECT * FROM employees
WHERE salary > 60000         -- greater than
  AND salary <= 100000       -- less than or equal
  AND dept_id != 3           -- not equal (also: <>)
  AND is_active = TRUE;

BETWEEN, IN, LIKE, IS NULL

SQL
-- BETWEEN (inclusive on both ends)
SELECT * FROM employees
WHERE salary BETWEEN 50000 AND 90000;

-- IN (matches any value in the list)
SELECT * FROM employees
WHERE dept_id IN (1, 2, 5);

-- LIKE (pattern matching)
-- %  = zero or more characters
-- _  = exactly one character
SELECT * FROM employees
WHERE first_name LIKE 'A%';     -- starts with A
WHERE email      LIKE '%@gmail.com';
WHERE last_name  LIKE '_upta';  -- Gupta, Supta …

-- NULL checks (never use = NULL!)
SELECT * FROM employees WHERE dept_id IS NULL;
SELECT * FROM employees WHERE dept_id IS NOT NULL;
🚨
NULL = NULL is always UNKNOWN in SQL, not TRUE. Always use IS NULL / IS NOT NULL.

Logical Operators

SQL
-- AND: both conditions must be true
WHERE salary > 70000 AND dept_id = 1

-- OR: at least one condition must be true
WHERE dept_id = 1 OR dept_id = 2

-- NOT: negates a condition
WHERE NOT (is_active = FALSE)

-- Combining — use parentheses for clarity!
WHERE (dept_id = 1 OR dept_id = 2)
  AND salary > 50000
  AND hire_date >= '2022-01-01';
07 ↕️

Sorting & Limiting Results

SQL
-- ORDER BY: ASC (default) or DESC
SELECT first_name, salary
FROM   employees
ORDER BY salary DESC, first_name ASC;

-- LIMIT: cap the number of rows returned
SELECT * FROM employees ORDER BY salary DESC LIMIT 10;

-- OFFSET: skip rows (for pagination)
-- Page 1: LIMIT 10 OFFSET 0
-- Page 2: LIMIT 10 OFFSET 10
-- Page 3: LIMIT 10 OFFSET 20
SELECT * FROM employees ORDER BY id LIMIT 10 OFFSET 20;

-- Top 5 highest-paid employees
SELECT first_name, last_name, salary
FROM   employees
ORDER BY salary DESC
LIMIT 5;

Aggregate Functions

Aggregate functions collapse many rows into a single summary value. They are used with GROUP BY.

FunctionDescriptionIgnores NULL?
COUNT(*)Counts all rows (including NULLs)No
COUNT(col)Counts non-NULL values in columnYes
SUM(col)Total sum of numeric columnYes
AVG(col)Arithmetic meanYes
MIN(col)Smallest valueYes
MAX(col)Largest valueYes
GROUP_CONCATConcatenates values into a string (MySQL)Yes
SQL
SELECT
    COUNT(*)             AS total_employees,
    COUNT(dept_id)       AS assigned_to_dept,
    SUM(salary)          AS payroll_total,
    AVG(salary)          AS avg_salary,
    MIN(salary)          AS lowest_salary,
    MAX(salary)          AS highest_salary,
    ROUND(AVG(salary), 2) AS avg_salary_rounded
FROM employees
WHERE is_active = TRUE;
09 📦

GROUP BY & HAVING

GROUP BY groups rows that share a column value so aggregates are computed per group. HAVING filters those groups (like WHERE but post-aggregation).

SQL
-- Headcount and avg salary per department
SELECT
    dept_id,
    COUNT(*)      AS headcount,
    AVG(salary)   AS avg_salary,
    SUM(salary)   AS total_payroll
FROM   employees
WHERE  is_active = TRUE        -- filter rows BEFORE grouping
GROUP BY dept_id
HAVING COUNT(*) > 5           -- filter groups AFTER aggregation
   AND AVG(salary) > 60000
ORDER BY avg_salary DESC;
💡
WHERE vs HAVING: WHERE filters individual rows before grouping. HAVING filters grouped results after aggregation. You cannot use aggregate functions in WHERE.
SQL
-- GROUP BY with ROLLUP (subtotals + grand total)
SELECT
    dept_id,
    SUM(salary) AS dept_payroll
FROM   employees
GROUP BY dept_id WITH ROLLUP;
-- Last row will have NULL dept_id = grand total

JOINS

JOINs combine rows from two or more tables based on a related column. This is one of the most important concepts in SQL.

Join Types Visualized

Join TypeReturnsUse When
INNER JOINMatching rows in BOTH tablesYou need data that exists in both sides
LEFT JOINAll left rows + matching rightKeep all left rows, even if no match
RIGHT JOINAll right rows + matching leftKeep all right rows (rarely used; prefer LEFT)
FULL OUTER JOINAll rows from both tablesFind all unmatched rows on either side
CROSS JOINCartesian product (every combination)Generate all possible pairs
SELF JOINA table joined to itselfHierarchical data (manager → employee)

INNER JOIN

SQL
-- Show each employee with their department name
SELECT
    e.first_name,
    e.last_name,
    e.salary,
    d.name AS department
FROM       employees   e
INNER JOIN departments d ON e.dept_id = d.id;

LEFT JOIN

SQL
-- All employees, even those without a department
SELECT
    e.first_name,
    COALESCE(d.name, 'Unassigned') AS department
FROM      employees   e
LEFT JOIN departments d ON e.dept_id = d.id;

-- Find employees with NO department (anti-join)
SELECT e.*
FROM      employees   e
LEFT JOIN departments d ON e.dept_id = d.id
WHERE     d.id IS NULL;

SELF JOIN

SQL
-- employees table has a manager_id column pointing to another employee
SELECT
    e.first_name              AS employee,
    m.first_name              AS manager
FROM      employees e
LEFT JOIN employees m ON e.manager_id = m.id;

Multi-Table JOIN

SQL
SELECT
    e.first_name,
    d.name        AS department,
    p.title       AS project,
    l.city        AS office_city
FROM       employees   e
INNER JOIN departments d  ON e.dept_id   = d.id
LEFT JOIN  emp_projects ep ON e.id        = ep.emp_id
LEFT JOIN  projects     p  ON ep.proj_id  = p.id
INNER JOIN locations    l  ON d.loc_id    = l.id;
11 🪆

Subqueries

A subquery is a query nested inside another query. They can appear in SELECT, FROM, WHERE, or HAVING.

Subquery in WHERE (Scalar)

SQL
-- Employees earning more than the company average
SELECT first_name, salary
FROM   employees
WHERE  salary > (
    SELECT AVG(salary) FROM employees
);

Subquery with IN

SQL
-- Employees in departments located in Mumbai
SELECT first_name
FROM   employees
WHERE  dept_id IN (
    SELECT id
    FROM   departments d
    JOIN   locations l ON d.loc_id = l.id
    WHERE  l.city = 'Mumbai'
);

Correlated Subquery

A correlated subquery references a column from the outer query. It runs once per row — can be slow on large datasets.

SQL
-- Employees earning more than their department's average
SELECT e.first_name, e.salary, e.dept_id
FROM   employees e
WHERE  e.salary > (
    SELECT AVG(e2.salary)
    FROM   employees e2
    WHERE  e2.dept_id = e.dept_id  -- ← references outer query
);

EXISTS & NOT EXISTS

SQL
-- Departments that HAVE at least one employee
SELECT d.name
FROM   departments d
WHERE EXISTS (
    SELECT 1 FROM employees e WHERE e.dept_id = d.id
);

-- Departments with NO employees
SELECT d.name
FROM   departments d
WHERE NOT EXISTS (
    SELECT 1 FROM employees e WHERE e.dept_id = d.id
);
12 ⚙️

Set Operators

Set operators combine the results of two or more SELECT queries. Both queries must have the same number of columns with compatible data types.

SQL
-- UNION: combine results, remove duplicates
SELECT email FROM customers
UNION
SELECT email FROM employees;

-- UNION ALL: keep duplicates (faster)
SELECT email FROM customers
UNION ALL
SELECT email FROM employees;

-- INTERSECT: only rows that appear in BOTH
SELECT email FROM customers
INTERSECT
SELECT email FROM employees;

-- EXCEPT / MINUS: rows in first but NOT in second
SELECT email FROM customers
EXCEPT
SELECT email FROM employees;

String Functions

SQL
SELECT
    LENGTH('Hello'),                    -- 5 (number of chars)
    UPPER('hello'),                     -- 'HELLO'
    LOWER('HELLO'),                     -- 'hello'
    TRIM('  hello  '),                  -- 'hello'

    REPEAT('ab', 3);                    -- 'ababab'

Date & Time Functions

SQL
SELECT
    NOW(),                                    -- 2025-01-15 14:30:00
    CURDATE(),                                -- 2025-01-15
    CURTIME(),                                -- 14:30:00
    YEAR(NOW()),                              -- 2025
    MONTH(NOW()),                             -- 1

15 🪟

Window Functions

Window functions perform calculations across a set of rows related to the current row — without collapsing rows like GROUP BY does. They're among the most powerful tools in SQL.

💡
Syntax: function() OVER (PARTITION BY col ORDER BY col ROWS BETWEEN ...)

Ranking Functions

SQL
SELECT
    first_name,
    dept_id,
    salary,
    -- ROW_NUMBER: unique sequential rank (no ties)
    ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS row_num,
    -- RANK: same rank for ties, then gaps
    RANK()       OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rank,
    -- DENSE_RANK: same rank for ties, NO gaps
    DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS dense_rank,
    -- NTILE: divide into N buckets (quartiles, deciles)
    NTILE(4)     OVER (ORDER BY salary) AS quartile
FROM employees;

Aggregate Window Functions

SQL
SELECT
    first_name,
    dept_id,
    salary,
    -- Running total within department
    SUM(salary) OVER (
        PARTITION BY dept_id
        ORDER BY hire_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_payroll,
    -- Dept average alongside each row
    AVG(salary) OVER (PARTITION BY dept_id) AS dept_avg,
    -- Difference from dept average
    salary - AVG(salary) OVER (PARTITION BY dept_id) AS diff_from_avg
FROM employees;

LAG & LEAD

SQL
-- Compare each month's sales to the previous month
SELECT
    sale_month,
    revenue,
    LAG(revenue, 1, 0) OVER (ORDER BY sale_month)  AS prev_month_rev,
    LEAD(revenue, 1, 0) OVER (ORDER BY sale_month) AS next_month_rev,
    revenue - LAG(revenue) OVER (ORDER BY sale_month) AS mom_change
FROM monthly_sales;

FIRST_VALUE & LAST_VALUE

SQL
SELECT
    first_name,
    dept_id,
    salary,
    FIRST_VALUE(first_name) OVER (
        PARTITION BY dept_id ORDER BY salary DESC
    ) AS highest_earner_in_dept
FROM employees;
16 🧩

Common Table Expressions (CTEs)

CTEs create named temporary result sets within a query. They improve readability, allow reuse, and enable recursion. Defined with the WITH keyword.

Basic CTE

SQL
WITH dept_stats AS (
    SELECT
        dept_id,
        AVG(salary)  AS avg_sal,
        COUNT(*)     AS headcount
    FROM   employees
    GROUP BY dept_id
)
SELECT
    e.first_name,
    e.salary,
    ds.avg_sal,
    e.salary - ds.avg_sal  AS diff
FROM  employees e
JOIN  dept_stats ds ON e.dept_id = ds.dept_id
WHERE e.salary > ds.avg_sal;

Multiple CTEs (Chained)

SQL
WITH
active_employees AS (
    SELECT * FROM employees WHERE is_active = TRUE
),
high_earners AS (
    SELECT * FROM active_employees WHERE salary > 80000
),
dept_summary AS (
    SELECT dept_id, COUNT(*) AS high_earner_count
    FROM   high_earners
    GROUP BY dept_id
)
SELECT d.name, ds.high_earner_count
FROM   departments d
JOIN   dept_summary ds ON d.id = ds.dept_id
ORDER BY ds.high_earner_count DESC;

Recursive CTE

Used for hierarchical data like org charts, folder structures, or bill-of-materials.

SQL
WITH RECURSIVE org_chart AS (
    -- Anchor: start with the CEO (no manager)
    SELECT id, first_name, manager_id, 0 AS level
    FROM   employees
    WHERE  manager_id IS NULL

    UNION ALL

    -- Recursive: find direct reports
    SELECT e.id, e.first_name, e.manager_id, oc.level + 1
    FROM   employees e
    JOIN   org_chart oc ON e.manager_id = oc.id
)
SELECT REPEAT('  ', level) || first_name AS hierarchy
FROM   org_chart
ORDER BY level;

Views

A view is a saved SQL query that behaves like a virtual table. It doesn't store data itself — it stores the query definition and executes it on demand.

SQL
-- Create a view
CREATE VIEW employee_details AS
    SELECT
        e.id,
        e.first_name,
        e.last_name,
        e.salary,
        d.name AS department
    FROM       employees   e
    INNER JOIN departments d ON e.dept_id = d.id
    WHERE      e.is_active = TRUE;

-- Use it like a table
SELECT * FROM employee_details WHERE department = 'Engineering';

-- Update the definition
CREATE OR REPLACE VIEW employee_details AS ...;

-- Remove a view
DROP VIEW employee_details;
Simple ViewBased on a single table with no aggregations — can be updated (DML allowed).
Complex ViewUses JOINs, GROUP BY, subqueries — usually read-only.
Materialized ViewPhysically stores the query result (PostgreSQL, Oracle). Needs periodic refresh. Very fast to read.
18 ⚙️

Stored Procedures & Functions

Stored procedures are precompiled SQL blocks stored in the database. They can have parameters, control flow (IF/WHILE), and execute complex logic.

Stored Procedure

SQL
DELIMITER $$

CREATE PROCEDURE GiveRaise(
    IN  p_dept_id  INT,
    IN  p_percent  DECIMAL(5,2),
    OUT p_affected INT
)
BEGIN
    UPDATE employees
    SET    salary = salary * (1 + p_percent/100)
    WHERE  dept_id   = p_dept_id
      AND  is_active = TRUE;

    SET p_affected = ROW_COUNT();
END$$

DELIMITER ;

-- Call the procedure
CALL GiveRaise(2, 10.00, @rows);
SELECT @rows AS employees_updated;

User-Defined Function

SQL
DELIMITER $$

CREATE FUNCTION AnnualSalary(p_salary DECIMAL(10,2))
RETURNS DECIMAL(12,2)
DETERMINISTIC
BEGIN
    RETURN p_salary * 12;
END$$

DELIMITER ;

-- Use it like a built-in function
SELECT first_name, AnnualSalary(salary) AS yearly
FROM   employees;
19 ⚡

Indexes

Indexes are data structures (usually B-Trees) that speed up data retrieval. Think of them like a book's index — instead of scanning every page, you jump directly to the right one.

✅ Without Index

Full table scan — reads every row to find matches. O(n) time. Fine for small tables, catastrophic for millions of rows.

⚡ With Index

B-tree lookup — jumps directly to matching rows. O(log n) time. Can make queries 100x–1000x faster.

SQL
-- Single column index
CREATE INDEX idx_emp_email     ON employees(email);
CREATE INDEX idx_emp_dept      ON employees(dept_id);

-- Composite index (column order matters!)
CREATE INDEX idx_dept_salary   ON employees(dept_id, salary);

-- Unique index (enforces uniqueness)
CREATE UNIQUE INDEX idx_emp_email_unique ON employees(email);

-- Full-text index (for LIKE '%...%' searches)
CREATE FULLTEXT INDEX idx_ft_name ON employees(first_name, last_name);

-- View indexes on a table
SHOW INDEXES FROM employees;

-- Drop an index
DROP INDEX idx_emp_dept ON employees;
⚠️
Index trade-offs: Indexes speed up SELECTs but slow down INSERT/UPDATE/DELETE (the index must be maintained). Don't index every column — index columns used in WHERE, JOIN, and ORDER BY clauses.

Index Best Practices

  • Index columns used in WHERE clauses and JOIN conditions
  • Index columns used in ORDER BY if sorting large result sets
  • For composite indexes, put the most selective column first
  • Avoid indexing low-cardinality columns (e.g. boolean flags)
  • Use EXPLAIN to see if queries are using indexes
  • Delete unused indexes — they waste space and slow writes
20 🔒

Transactions & ACID

A transaction is a group of SQL operations that must all succeed or all fail together — ensuring data integrity.

ACID Properties

AtomicityAll operations in a transaction succeed, or none do. No partial updates.
ConsistencyThe database moves from one valid state to another. All constraints are respected.
IsolationConcurrent transactions don't interfere with each other. Different isolation levels control this.
DurabilityCommitted changes persist even after a crash. Written to disk / write-ahead log.

Transaction Syntax

SQL
-- Bank transfer: debit one account, credit another
START TRANSACTION;

UPDATE accounts SET balance = balance - 5000 WHERE id = 101;
UPDATE accounts SET balance = balance + 5000 WHERE id = 202;

-- Check: did both updates succeed?
SELECT balance FROM accounts WHERE id IN (101, 202);

-- All good → save changes permanently
COMMIT;

-- OR if something went wrong → undo everything
ROLLBACK;

-- SAVEPOINT: partial rollback within a transaction
SAVEPOINT before_transfer;
-- ... do some work ...
ROLLBACK TO SAVEPOINT before_transfer;

Isolation Levels

LevelDirty ReadNon-Repeatable ReadPhantom Read
READ UNCOMMITTED✅ Possible✅ Possible✅ Possible
READ COMMITTED❌ Prevented✅ Possible✅ Possible
REPEATABLE READ❌ Prevented❌ Prevented✅ Possible
SERIALIZABLE❌ Prevented❌ Prevented❌ Prevented
💡
MySQL InnoDB default is REPEATABLE READ. PostgreSQL default is READ COMMITTED. Higher isolation = safer but slower.
21 🚀

Query Optimization

Writing correct SQL is step one. Writing fast SQL requires understanding how the database executes queries.

EXPLAIN — Query Execution Plan

SQL
-- See how MySQL will execute a query
EXPLAIN SELECT * FROM employees WHERE email = 'priya@company.com';

-- More detailed output (MySQL 8+)
EXPLAIN ANALYZE SELECT * FROM employees e
JOIN departments d ON e.dept_id = d.id
WHERE e.salary > 80000;

Key EXPLAIN columns to watch: type (ALL is bad, ref/eq_ref is good), rows (estimated rows scanned), key (index used).

Optimization Rules

  • SELECT only what you need — avoid SELECT * in production; fetch only required columns
  • Use indexes on WHERE, JOIN, and ORDER BY columns
  • Avoid functions on indexed columns in WHERE — WHERE YEAR(hire_date) = 2023 can't use an index on hire_date; use WHERE hire_date BETWEEN '2023-01-01' AND '2023-12-31' instead
  • Use EXISTS instead of IN for large subquery sets
  • Prefer JOINs over correlated subqueries — correlated subqueries run once per row
  • Use LIMIT when you only need a subset of rows
  • Avoid OR on different columns — it can prevent index use; consider UNION instead
  • Avoid leading wildcards — LIKE '%something' can't use a B-tree index
  • Batch large DML operations — deleting 1M rows at once locks the table; delete in chunks
  • Analyze & update statistics — ANALYZE TABLE employees keeps query optimizer estimates accurate

Normalization

Normalization is the process of organizing database tables to reduce redundancy and improve data integrity.

1NF First Normal Form — Each column holds atomic (indivisible) values. No repeating groups. Each row is unique (has a primary key). Example: splitting "John, Jane" in one cell into separate rows.
2NF Second Normal Form — Must be in 1NF. Every non-key column is fully dependent on the entire primary key (removes partial dependencies). Applies when PK is composite.
3NF Third Normal Form — Must be in 2NF. No transitive dependencies — non-key columns depend only on the PK, not on other non-key columns. Example: storing city in employee table when city depends on zip code (not employee ID).
BCNF Boyce-Codd NF — Stronger version of 3NF. Every functional dependency X → Y, X must be a superkey.
Denormalization Intentionally adding redundancy for read performance in analytics/reporting. Common in data warehouses. Trade integrity for speed.
✅
Aim for 3NF in OLTP systems (applications). Consider denormalization only in analytics/reporting (OLAP) where read speed is critical.

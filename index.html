What is SQL? Foundation

SQL (Structured Query Language) is the standard language for interacting with relational databases. It allows you to create, read, update, and delete data (CRUD). SQL is used in almost every application that stores structured data — banking systems, e-commerce platforms, social networks, and more.

Key SQL dialects include MySQL, PostgreSQL, SQL Server (T-SQL), Oracle, and SQLite. The syntax is mostly standard, with small differences per dialect.

DDL

Data Definition Language — CREATE, ALTER, DROP, TRUNCATE

DML

Data Manipulation Language — SELECT, INSERT, UPDATE, DELETE

DCL

Data Control Language — GRANT, REVOKE, COMMIT, ROLLBACK

CREATE TABLE — Building Structure DDL

CREATE TABLE defines a new table with columns, data types, and constraints. This is the foundation of your database schema.

SQL
CREATE TABLE employees (
    employee_id   INT           PRIMARY KEY AUTO_INCREMENT,
    first_name    VARCHAR(50)   NOT NULL,
    last_name     VARCHAR(50)   NOT NULL,
    email         VARCHAR(100)  UNIQUE NOT NULL,
    salary        DECIMAL(10,2) DEFAULT 50000.00,
    department_id INT,
    hire_date     DATE           DEFAULT CURRENT_DATE,
    FOREIGN KEY (department_id) REFERENCES departments(dept_id)
);
ℹ
Constraints explained: PRIMARY KEY = unique identifier per row. NOT NULL = value required. UNIQUE = no duplicates. DEFAULT = fallback value. FOREIGN KEY = links to another table ensuring referential integrity.
SELECT — Retrieving Data DML

Basic SELECT Syntax

SELECT is the most used SQL command. It retrieves rows from one or more tables. The execution order is: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT.

-- Select all columns
SELECT * FROM employees;

-- Select specific columns
SELECT first_name, last_name, salary
FROM employees;

-- With alias for readability
SELECT
    first_name AS 'First Name',
    salary * 12 AS annual_salary
FROM employees;

WHERE Clause — Filtering Rows

WHERE filters rows based on one or more conditions. Only rows matching the condition are returned.

-- Single condition
SELECT * FROM employees
WHERE salary > 60000;

-- Multiple conditions using AND / OR
SELECT * FROM employees
WHERE department_id = 3
  AND salary BETWEEN 50000 AND 80000;

-- IN operator — match any value in list
SELECT * FROM employees
WHERE department_id IN (1, 3, 5);

-- LIKE — pattern matching  (% = any chars, _ = single char)
SELECT * FROM employees
WHERE email LIKE '%@gmail.com';

-- IS NULL — check for missing values
SELECT * FROM employees
WHERE department_id IS NULL;

ORDER BY, LIMIT, DISTINCT

-- Sort by salary descending
SELECT first_name, salary
FROM employees
ORDER BY salary DESC;

-- Top 5 highest paid
SELECT first_name, salary
FROM employees
ORDER BY salary DESC
LIMIT 5;

-- Distinct values only (removes duplicates)
SELECT DISTINCT department_id
FROM employees;
INSERT, UPDATE, DELETE DML
-- INSERT single row
INSERT INTO employees (first_name, last_name, email, salary)
VALUES ('Rahul', 'Sharma', 'rahul@example.com', 75000);

-- INSERT multiple rows
INSERT INTO employees (first_name, last_name, email)
VALUES
    ('Priya', 'Patel', 'priya@example.com'),
    ('Amit', 'Kumar', 'amit@example.com');

-- UPDATE — always use WHERE to avoid updating all rows!
UPDATE employees
SET salary = salary * 1.10,
    last_name = 'Singh'
WHERE employee_id = 5;

-- DELETE — always use WHERE!
DELETE FROM employees
WHERE employee_id = 5;
⚠
Critical Rule: Always use a WHERE clause with UPDATE and DELETE. Without it, you modify or delete ALL rows in the table — a very common and costly mistake!
Aggregate Functions Core

Aggregate functions compute a single value from multiple rows. They are used with GROUP BY to calculate statistics per group.

-- COUNT, SUM, AVG, MIN, MAX
SELECT
    COUNT(*) AS total_employees,
    SUM(salary) AS total_payroll,
    AVG(salary) AS avg_salary,
    MIN(salary) AS lowest_salary,
    MAX(salary) AS highest_salary
FROM employees;

-- GROUP BY — compute stats per department
SELECT
    department_id,
    COUNT(*) AS headcount,
    AVG(salary) AS avg_salary
FROM employees
GROUP BY department_id;

-- HAVING — filter groups (like WHERE but for groups)
SELECT department_id, AVG(salary) AS avg_sal
FROM employees
GROUP BY department_id
HAVING AVG(salary) > 60000;
✦
WHERE vs HAVING: WHERE filters rows before grouping. HAVING filters groups after GROUP BY. You cannot use aggregate functions in WHERE — use HAVING for that.
JOINs — Combining Tables Intermediate

JOINs combine rows from two or more tables based on a related column. Understanding JOINs is essential — most real-world queries involve multiple tables.

INNER JOIN — Only Matching Rows

Returns rows where there is a match in both tables. Rows without a match are excluded from results.

SELECT
    e.first_name,
    e.salary,
    d.department_name
FROM employees e
INNER JOIN departments d
    ON e.department_id = d.dept_id;
-- Only employees who HAVE a department are returned

LEFT JOIN — All Left + Matching Right

Returns all rows from the left table, and matching rows from the right. Where no match exists, NULL values fill in the right side.

SELECT
    e.first_name,
    d.department_name  -- NULL if no dept assigned
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.dept_id;
-- ALL employees shown, even those without a department

RIGHT JOIN, FULL OUTER JOIN, SELF JOIN, CROSS JOIN

-- RIGHT JOIN — all departments, even with no employees
SELECT e.first_name, d.department_name
FROM employees e
RIGHT JOIN departments d ON e.department_id = d.dept_id;

-- FULL OUTER JOIN — all rows from both sides
SELECT e.first_name, d.department_name
FROM employees e
FULL OUTER JOIN departments d ON e.department_id = d.dept_id;

-- SELF JOIN — compare rows within the same table
-- Find employees and their manager's name
SELECT
    e.first_name AS employee,
    m.first_name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.employee_id;

-- CROSS JOIN — every combination (Cartesian product)
SELECT colors.name, sizes.label
FROM colors
CROSS JOIN sizes; -- 3 colors × 5 sizes = 15 rows
Subqueries Intermediate

A subquery is a query nested inside another query. It can appear in SELECT, FROM, WHERE, and HAVING clauses. Subqueries are powerful but sometimes slower than JOINs.

-- WHERE subquery: employees earning above average
SELECT first_name, salary
FROM employees
WHERE salary > (
    SELECT AVG(salary) FROM employees
);

-- FROM subquery (derived table)
SELECT dept, avg_sal
FROM (
    SELECT department_id AS dept, AVG(salary) AS avg_sal
    FROM employees
    GROUP BY department_id
) AS dept_avg
WHERE avg_sal > 65000;

-- EXISTS — check if subquery returns any rows
SELECT d.department_name
FROM departments d
WHERE EXISTS (
    SELECT 1 FROM employees e
    WHERE e.department_id = d.dept_id
);

-- Correlated subquery — references outer query per row
SELECT first_name, salary, department_id
FROM employees e
WHERE salary = (
    SELECT MAX(salary)
    FROM employees
    WHERE department_id = e.department_id
); -- Top earner in each department
CTEs (Common Table Expressions) Intermediate

CTEs (WITH clause) create temporary named result sets that you can reference in a query. They make complex queries more readable, and enable recursive queries.

-- Basic CTE
WITH dept_stats AS (
    SELECT
        department_id,
        COUNT(*) AS headcount,
        AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department_id
)
SELECT d.department_name, ds.headcount, ds.avg_salary
FROM dept_stats ds
JOIN departments d ON ds.department_id = d.dept_id
ORDER BY ds.avg_salary DESC;

-- Multiple CTEs chained together
WITH
top_earners AS (
    SELECT * FROM employees WHERE salary > 80000
),
their_depts AS (
    SELECT DISTINCT department_id FROM top_earners
)
SELECT department_name
FROM departments
WHERE dept_id IN (SELECT * FROM their_depts);
String & Date Functions Intermediate

String Functions

SELECT
    UPPER(first_name),              -- 'RAHUL'
    LOWER(last_name),               -- 'sharma'
    CONCAT(first_name,' ',last_name), -- 'Rahul Sharma'
    LENGTH(email),                  -- character count
    TRIM(first_name),               -- removes leading/trailing spaces
    SUBSTRING(email, 1, 5),         -- first 5 chars
    REPLACE(email, 'gmail', 'yahoo'), -- replace substring
    INSTR(email, '@')               -- position of '@'
FROM employees;

Date Functions

SELECT
    NOW(),                               -- current datetime
    CURDATE(),                           -- today's date
    YEAR(hire_date),                     -- extract year
    MONTH(hire_date),                    -- extract month
    DATEDIFF(CURDATE(), hire_date),    -- days since hired
    DATE_ADD(hire_date, INTERVAL 1 YEAR), -- add 1 year
    DATE_FORMAT(hire_date, '%d/%m/%Y') -- format date
FROM employees;
UNION, INTERSECT, EXCEPT Intermediate
-- UNION — combine results from two queries (removes duplicates)
SELECT email FROM customers
UNION
SELECT email FROM employees;

-- UNION ALL — keeps duplicates (faster)
SELECT email FROM customers
UNION ALL
SELECT email FROM employees;

-- INTERSECT — rows present in BOTH queries
SELECT email FROM customers
INTERSECT
SELECT email FROM employees;

-- EXCEPT — rows in first query but NOT in second
SELECT email FROM customers
EXCEPT
SELECT email FROM employees;
ℹ
Rule: All queries in a UNION must have the same number of columns and compatible data types. Column names come from the first query.
Window Functions Advanced

Window functions perform calculations across a set of rows related to the current row — without collapsing them into a single row like GROUP BY does. They are among the most powerful SQL features.

ℹ
Syntax: function() OVER (PARTITION BY col ORDER BY col ROWS/RANGE frame)
-- ROW_NUMBER — unique sequential rank per partition
SELECT
    first_name,
    department_id,
    salary,
    ROW_NUMBER() OVER (
        PARTITION BY department_id
        ORDER BY salary DESC
    ) AS rank_in_dept
FROM employees;

-- RANK — same value gets same rank, skips next number
-- DENSE_RANK — same value gets same rank, no skipping
SELECT
    first_name, salary,
    RANK()       OVER (ORDER BY salary DESC) AS rnk,
    DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rnk
FROM employees;

-- LAG / LEAD — access previous/next row values
SELECT
    month, revenue,
    LAG(revenue, 1) OVER (ORDER BY month) AS prev_month_rev,
    revenue - LAG(revenue, 1) OVER (ORDER BY month) AS mom_change
FROM monthly_sales;

-- SUM / AVG as running total / moving average
SELECT
    order_date, amount,
    SUM(amount) OVER (ORDER BY order_date) AS running_total,
    AVG(amount) OVER (
        ORDER BY order_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS rolling_7day_avg
FROM orders;

-- NTILE — divide rows into N equal buckets
SELECT
    first_name, salary,
    NTILE(4) OVER (ORDER BY salary) AS salary_quartile
FROM employees;
Recursive CTEs Advanced

Recursive CTEs reference themselves, allowing you to work with hierarchical or tree-structured data like org charts, folder structures, or bill-of-materials.

-- Employee org chart hierarchy
WITH RECURSIVE org_chart AS (
    -- Anchor: start with the CEO (no manager)
    SELECT employee_id, first_name, manager_id, 0 AS level
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- Recursive: find direct reports of each found employee
    SELECT e.employee_id, e.first_name, e.manager_id, oc.level + 1
    FROM employees e
    INNER JOIN org_chart oc ON e.manager_id = oc.employee_id
)
SELECT REPEAT('  ', level) || first_name AS hierarchy, level
FROM org_chart
ORDER BY level, first_name;
Indexes Advanced

An index is a data structure (usually a B-tree) that dramatically speeds up data retrieval at the cost of extra storage and slightly slower writes. Think of it like a book's index — instead of reading every page, you jump to the right one.

-- Create a simple index
CREATE INDEX idx_emp_salary ON employees(salary);

-- Unique index (like UNIQUE constraint)
CREATE UNIQUE INDEX idx_emp_email ON employees(email);

-- Composite index (for multi-column queries)
CREATE INDEX idx_dept_salary ON employees(department_id, salary);
-- Useful for: WHERE department_id = X AND salary > Y

-- Drop an index
DROP INDEX idx_emp_salary ON employees;
⚠
Index Guidelines: Index columns used in WHERE, JOIN, and ORDER BY. Don't over-index — each index slows INSERT/UPDATE/DELETE. The leftmost column in a composite index must be in your WHERE clause for it to be used.
Stored Procedures & Functions Advanced

Stored Procedure

A stored procedure is a reusable block of SQL code stored in the database. It can accept parameters, execute complex logic, and perform multiple operations.

DELIMITER //
CREATE PROCEDURE GiveRaise(
    IN p_dept_id   INT,
    IN p_percent   DECIMAL(5,2),
    OUT p_affected INT
)
BEGIN
    UPDATE employees
    SET salary = salary * (1 + p_percent / 100)
    WHERE department_id = p_dept_id;

    SET p_affected = ROW_COUNT();
END//
DELIMITER ;

-- Call the procedure
CALL GiveRaise(3, 10.0, @rows_updated);
SELECT @rows_updated;

User-Defined Function (UDF)

A function always returns a single value and can be used inline in queries, unlike stored procedures.

DELIMITER //
CREATE FUNCTION annual_salary(monthly_sal DECIMAL(10,2))
RETURNS DECIMAL(12,2)
DETERMINISTIC
BEGIN
    RETURN monthly_sal * 12;
END//
DELIMITER ;

-- Use the function in a query
SELECT first_name, annual_salary(salary) AS yearly
FROM employees;
Transactions Advanced

A transaction is a unit of work that either fully succeeds or fully fails. Transactions follow ACID properties: Atomicity, Consistency, Isolation, Durability.

-- Bank transfer: debit one account, credit another
START TRANSACTION;

UPDATE accounts
SET balance = balance - 5000
WHERE account_id = 101;

UPDATE accounts
SET balance = balance + 5000
WHERE account_id = 202;

-- If both succeed:
COMMIT;

-- If anything fails (in application code):
ROLLBACK;

-- SAVEPOINT — partial rollback within transaction
SAVEPOINT after_first_update;
-- ... do more work ...
ROLLBACK TO SAVEPOINT after_first_update;
✦
ACID explained: Atomicity — all or nothing. Consistency — data rules always maintained. Isolation — concurrent transactions don't interfere. Durability — committed data survives crashes.
Views Advanced

A view is a saved SQL query that acts like a virtual table. It simplifies complex queries, enforces security by hiding certain columns, and centralizes business logic.

-- Create a view for HR reporting
CREATE VIEW employee_summary AS
    SELECT
        e.employee_id,
        CONCAT(e.first_name, ' ', e.last_name) AS full_name,
        d.department_name,
        e.salary
    FROM employees e
    JOIN departments d ON e.department_id = d.dept_id;

-- Query the view just like a table
SELECT * FROM employee_summary
WHERE department_name = 'Engineering';

-- Drop view
DROP VIEW employee_summary;

Query Optimization Techniques Performance

Index Optimization

-- BAD: Function on indexed column prevents index use
SELECT * FROM orders
WHERE YEAR(order_date) = 2024; -- Full table scan!

-- GOOD: Use range on the column directly
SELECT * FROM orders
WHERE order_date BETWEEN '2024-01-01' AND '2024-12-31';

-- BAD: Wildcard at start prevents index use
WHERE email LIKE '%gmail%'; -- Full scan

-- GOOD: Wildcard only at end (prefix search)
WHERE email LIKE 'rahul%'; -- Uses index

SELECT Specific Columns

-- BAD: SELECT * fetches all data — more I/O
SELECT * FROM orders WHERE customer_id = 42;

-- GOOD: Only fetch what you need
SELECT order_id, order_date, total_amount
FROM orders
WHERE customer_id = 42;

Use EXISTS instead of IN for large sets

-- BAD for large subquery result sets
SELECT * FROM customers
WHERE customer_id IN (
    SELECT customer_id FROM orders
    WHERE order_date > '2024-01-01'
);

-- GOOD: EXISTS short-circuits after first match
SELECT * FROM customers c
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.customer_id = c.customer_id
      AND o.order_date > '2024-01-01'
);

Pagination — OFFSET vs Keyset

-- BAD: OFFSET pagination (slow on large tables)
SELECT * FROM orders
ORDER BY order_id
LIMIT 20 OFFSET 10000; -- scans 10020 rows to return 20

-- GOOD: Keyset/cursor pagination (always fast)
SELECT * FROM orders
WHERE order_id > 10000  -- last seen ID
ORDER BY order_id
LIMIT 20;
Normalization vs Denormalization Performance

Normalization ✓

Less data redundancy. Faster writes (INSERT/UPDATE). Better for OLTP systems. Harder to query (more JOINs needed).

Denormalization ✓

Faster reads (fewer JOINs). More redundancy. Better for OLAP/reporting. Updates must modify multiple places.

Frequently Asked Interview Questions Interview
Q1 What is the difference between WHERE and HAVING? ▼

WHERE filters rows before any grouping or aggregation happens. It works on individual rows and cannot use aggregate functions.

HAVING filters groups after the GROUP BY clause has been applied. It can use aggregate functions like COUNT(), SUM(), AVG().

SELECT dept_id, COUNT(*) AS total
FROM employees
WHERE salary > 40000          -- filter rows first
GROUP BY dept_id
HAVING COUNT(*) > 5;           -- then filter groups

Memory trick: WHERE works on rows, HAVING works on groups.

Q2 What is the difference between DELETE, TRUNCATE, and DROP? ▼

DELETE: Removes specific rows based on a WHERE clause. It is a DML command, logs each deletion, can be rolled back in a transaction, and fires triggers. Slow on large tables.

TRUNCATE: Removes ALL rows from a table instantly. It is a DDL command, minimal logging, cannot be rolled back in most databases, resets auto-increment counter. Very fast.

DROP: Completely removes the table structure AND all its data from the database. Cannot be undone (without backup).

DELETE FROM orders WHERE status = 'cancelled'; -- specific rows
TRUNCATE TABLE temp_logs;    -- all rows, table stays
DROP TABLE old_archive;       -- table gone completely
Q3 Explain the difference between RANK(), DENSE_RANK(), and ROW_NUMBER() ▼

All three assign a number to each row within a partition, but they handle ties differently:

ROW_NUMBER(): Always gives unique numbers (1,2,3,4,...). Ties get arbitrary but unique ranks.

RANK(): Tied rows get the same rank, but the next rank skips. Salaries 100,100,80 → ranks 1,1,3 (rank 2 is skipped).

DENSE_RANK(): Tied rows get the same rank, but the next rank does NOT skip. Salaries 100,100,80 → ranks 1,1,2 (no gap).

-- salary: 100, 100, 80, 70
ROW_NUMBER: 1, 2, 3, 4    -- always unique
RANK:       1, 1, 3, 4    -- gap after tie
DENSE_RANK: 1, 1, 2, 3    -- no gap after tie

Use ROW_NUMBER for pagination, RANK/DENSE_RANK for leaderboards.

Q4 What are the ACID properties of a transaction? ▼

A — Atomicity: All operations in a transaction either ALL succeed or ALL fail. If the bank debit succeeds but the credit fails, the debit is rolled back. No partial transactions.

C — Consistency: A transaction brings the database from one valid state to another valid state. All rules, constraints, and cascades are maintained after every transaction.

I — Isolation: Concurrent transactions execute as if they were serial. Changes made by one transaction are not visible to others until committed (depending on isolation level).

D — Durability: Once a transaction is committed, it is permanently saved even if the system crashes immediately after. The commit writes to disk (WAL logs).

Q5 What is a Primary Key vs a Foreign Key? ▼

Primary Key (PK): A column (or combination) that uniquely identifies each row in a table. It must be unique and NOT NULL. A table can only have one primary key. It creates a clustered index automatically.

Foreign Key (FK): A column in one table that references the primary key of another table. It enforces referential integrity — you cannot insert a FK value that doesn't exist in the referenced table, and you cannot delete a referenced PK row if dependent FK rows exist.

CREATE TABLE orders (
    order_id    INT PRIMARY KEY,    -- PK
    customer_id INT,
    FOREIGN KEY (customer_id)       -- FK
        REFERENCES customers(customer_id)
        ON DELETE CASCADE           -- auto-delete orders if customer deleted
);
Q6 What is a correlated subquery? How does it differ from a regular subquery? ▼

Regular subquery: Executes once independently and passes its result to the outer query. It has no reference to the outer query.

Correlated subquery: References a column from the outer query, so it executes once per row in the outer query. This makes it potentially slow for large tables but very powerful.

-- Regular: runs once
WHERE salary > (SELECT AVG(salary) FROM employees)

-- Correlated: runs once per outer row — finds each dept's max earner
WHERE salary = (
    SELECT MAX(salary) FROM employees inner_e
    WHERE inner_e.department_id = outer_e.department_id
)
Q7 What are indexes and when should you use them? ▼

An index is a separate data structure (usually a B-tree) that stores a sorted copy of one or more columns, with pointers back to the actual rows. This allows the database to find rows without scanning every row.

Use indexes on: Columns in WHERE clauses, JOIN conditions, ORDER BY columns, and columns with high cardinality (many unique values like email, user_id).

Avoid over-indexing: Every index slows down INSERT/UPDATE/DELETE because the index must be updated too. On a write-heavy table, fewer indexes is better.

Composite indexes: Follow the leftmost-prefix rule — a composite index on (a, b, c) can be used for queries on (a), (a,b), or (a,b,c), but NOT on just (b) or (c).

Q8 What is the difference between INNER JOIN and LEFT JOIN? ▼

INNER JOIN: Returns only the rows where there is a matching value in both tables. Rows without a match in either table are excluded completely.

LEFT JOIN: Returns all rows from the left table, regardless of whether there's a match in the right table. Where no match exists, the right side columns appear as NULL.

Use INNER JOIN when you only want complete matches. Use LEFT JOIN when you want all records from the primary table and optionally related data from another (e.g., customers who may or may not have placed orders).

-- INNER: only employees WITH a department
FROM employees e INNER JOIN departments d ON e.dept_id = d.id

-- LEFT: ALL employees, dept_name = NULL if unassigned
FROM employees e LEFT JOIN departments d ON e.dept_id = d.id
Q9 What is a CTE and what are its advantages over subqueries? ▼

A CTE (Common Table Expression) defined with the WITH clause creates a named temporary result set for the duration of one query. Advantages over subqueries include: better readability and modularity, ability to reference the CTE multiple times in the same query (a subquery would need to be repeated), support for recursion, and easier debugging by testing each CTE block independently.

-- CTE referenced twice without repeating logic
WITH high_earners AS (
    SELECT * FROM employees WHERE salary > 80000
)
SELECT COUNT(*) FROM high_earners     -- reference 1
UNION ALL
SELECT AVG(salary) FROM high_earners;  -- reference 2
Q10 What is normalization? Explain 1NF, 2NF, and 3NF with examples. ▼

1NF (First Normal Form): Every column must contain atomic (indivisible) values, and every row must be unique. Problem: A column like "phones: 9876,1234" violates 1NF. Fix: Create a separate phone_numbers table.

2NF (Second Normal Form): Must be in 1NF, and every non-key column must depend on the ENTIRE primary key (no partial dependency). Problem: In an order_items table with PK (order_id, product_id), if product_name depends only on product_id (not the full PK), that's a partial dependency. Fix: Move product_name to a products table.

3NF (Third Normal Form): Must be in 2NF, and no non-key column should depend on another non-key column (no transitive dependency). Problem: employee table has dept_id, dept_name, dept_location — dept_name and dept_location depend on dept_id, not on employee_id. Fix: Move them to a departments table.

Real-World Problem Solving Scenarios
01
Find the 2nd Highest Salary
Easy

Problem

Write a query to find the second highest salary from the employees table. Return NULL if no such salary exists.

Approach 1 — Using LIMIT/OFFSET

SELECT MAX(salary) AS second_highest
FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);

Approach 2 — Using DENSE_RANK (handles ties correctly)

SELECT DISTINCT salary
FROM (
    SELECT salary,
           DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
    FROM employees
) ranked
WHERE rnk = 2;

Why Approach 2 is better

If two people have the same highest salary, Approach 1 would skip to the third distinct value. DENSE_RANK ensures we correctly identify "second place" even when there are ties at the top.

02
Find Duplicate Records
Easy

Problem

Find all duplicate email addresses in the customers table — emails that appear more than once.

Solution

-- Find emails that appear more than once
SELECT email, COUNT(*) AS occurrences
FROM customers
GROUP BY email
HAVING COUNT(*) > 1;

-- See the full duplicate rows
SELECT * FROM customers
WHERE email IN (
    SELECT email
    FROM customers
    GROUP BY email
    HAVING COUNT(*) > 1
);

-- Delete duplicates, keep the one with lowest ID
DELETE FROM customers
WHERE customer_id NOT IN (
    SELECT MIN(customer_id)
    FROM customers
    GROUP BY email
);
03
Customers Who Never Placed an Order
Medium

Problem

Find all customers who have never placed an order. This is a classic "find missing relationship" query.

Approach 1 — LEFT JOIN + IS NULL

SELECT c.customer_id, c.name, c.email
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;

Approach 2 — NOT EXISTS

SELECT customer_id, name
FROM customers c
WHERE NOT EXISTS (
    SELECT 1 FROM orders o
    WHERE o.customer_id = c.customer_id
);

Which to use?

Both work. LEFT JOIN + IS NULL is more readable. NOT EXISTS often performs better on large datasets because it short-circuits on the first match found.

04
Running Total & Monthly Growth
Medium

Problem

Given a sales table with monthly revenue, calculate the running total and month-over-month growth percentage.

WITH monthly_data AS (
    SELECT
        DATE_FORMAT(order_date, '%Y-%m') AS month,
        SUM(amount) AS revenue
    FROM orders
    GROUP BY DATE_FORMAT(order_date, '%Y-%m')
)
SELECT
    month,
    revenue,
    SUM(revenue) OVER (ORDER BY month) AS running_total,
    LAG(revenue) OVER (ORDER BY month)  AS prev_revenue,
    ROUND(
        (revenue - LAG(revenue) OVER (ORDER BY month))
        / LAG(revenue) OVER (ORDER BY month) * 100,
    2) AS growth_pct
FROM monthly_data
ORDER BY month;

Key concepts used

CTE for clean aggregation, SUM() OVER() for running total (no partition = cumulative across all rows), LAG() to access previous row without a self-join.

Employees Earning More Than Their Manager
Medium

Problem

Find all employees who earn a higher salary than their direct manager.

SELECT
    e.first_name  AS employee,
    e.salary      AS emp_salary,
    m.first_name  AS manager,
    m.salary      AS mgr_salary
FROM employees e
INNER JOIN employees m
    ON e.manager_id = m.employee_id
WHERE e.salary > m.salary;

Explanation

This uses a SELF JOIN — joining the employees table to itself, aliasing it as 'e' (employee) and 'm' (manager). We link them through the manager_id column, then simply compare salaries in the WHERE clause.

06
Top N Products Per Category
Hard

Problem

Find the top 3 best-selling products in each category based on total revenue.

WITH product_revenue AS (
    -- Step 1: Calculate total revenue per product
    SELECT
        p.product_id,
        p.product_name,
        p.category,
        SUM(oi.quantity * oi.unit_price) AS total_revenue
    FROM products p
    JOIN order_items oi ON p.product_id = oi.product_id
    GROUP BY p.product_id, p.product_name, p.category
),
ranked_products AS (
    -- Step 2: Rank within each category
    SELECT
        *,
        DENSE_RANK() OVER (
            PARTITION BY category
            ORDER BY total_revenue DESC
        ) AS rank_in_cat
    FROM product_revenue
)
-- Step 3: Filter to top 3 per category
SELECT category, product_name, total_revenue, rank_in_cat
FROM ranked_products
WHERE rank_in_cat <= 3
ORDER BY category, rank_in_cat;

Problem-solving approach

Break complex problems into steps using CTEs. Step 1 aggregates raw data. Step 2 applies window ranking within partitions. Step 3 filters. This pattern — aggregate → rank → filter — solves most "top N per group" problems.

07
Pivot: Monthly Sales Report
Hard

Problem

Transform row data into columns — show sales for Jan, Feb, Mar as separate columns in one row per product.

-- Conditional aggregation pivot (works in all SQL databases)
SELECT
    product_name,
    SUM(CASE WHEN MONTH(sale_date) = 1 THEN amount ELSE 0 END) AS jan,
    SUM(CASE WHEN MONTH(sale_date) = 2 THEN amount ELSE 0 END) AS feb,
    SUM(CASE WHEN MONTH(sale_date) = 3 THEN amount ELSE 0 END) AS mar,
    SUM(amount) AS total
FROM sales
WHERE YEAR(sale_date) = 2024
GROUP BY product_name
ORDER BY total DESC;

Explanation

CASE WHEN inside SUM is the standard way to pivot in SQL. For each row, CASE returns the amount if it's the right month, otherwise 0. SUM then totals across all rows per product. This works in MySQL, PostgreSQL, and SQLite — unlike SQL Server's PIVOT operator which is not universal.

Query Execution Order

SQL executes clauses in this order — not the order you write them. Understanding this explains why you can't use SELECT aliases in WHERE:

1. FROM        -- identify tables
2. JOIN        -- combine tables
3. WHERE       -- filter rows
4. GROUP BY    -- group rows
5. HAVING      -- filter groups
6. SELECT      -- choose columns
7. DISTINCT    -- remove duplicates
8. ORDER BY    -- sort results
9. LIMIT       -- restrict rows

Performance Quick Wins

  • Add indexes on columns used in WHERE, JOIN ON, and ORDER BY
  • Use EXPLAIN to see if your index is being used
  • Avoid SELECT * — specify only needed columns
  • Avoid functions on indexed columns in WHERE (use range instead)
  • Use EXISTS instead of IN for large subquery results
  • Prefer keyset pagination over OFFSET for large datasets
  • Use LIMIT to test queries on large tables during development
  • Avoid DISTINCT when GROUP BY achieves the same result
  • Use connection pooling in applications
  • Partition large tables by date range for archive strategies
